     3|## 一、Redis vs Cassandra 数据差异（已修正理解）
     4|
     5|```
     6|                    ┌──────────────────────────────────┐
     7|                    │         Redis (L1, 24h)          │
     8|                    │  原始交易数据 (Raw Transaction)    │
     9|                    │  ──────────────────────────────  │
    10|                    │  • 最近24小时每笔交易的原始记录     │
    11|                    │  • 交易ID → {amount, device, IP,  │
    12|                    │    geo, timestamp, merchant...}   │
    13|                    │  • 账户 → 最近N笔交易列表          │
    14|                    │  • 设备ID → 关联账户列表           │
    15|                    │  TTL: 24h 自动过期                 │
    16|                    └──────────────────────────────────┘
    17|                              │
    18|                              │ 夜间Batch聚合
    19|                              ▼
    20|                    ┌──────────────────────────────────┐
    21|                    │      Cassandra (L2, 30-90d)       │
    22|                    │  预计算指标数据 (Pre-computed      │
    23|                    │  Aggregated Metrics)              │
    24|                    │  ──────────────────────────────  │
    25|                    │  • 30天交易总笔数、总金额           │
    26|                    │  • 日均/周均交易频率                │
    27|                    │  • 金额 99p/95p/50p 分位数         │
    28|                    │  • 近30天商户多样性指数             │
    29|                    │  • 近30天设备变化次数               │
    30|                    │  • 历史 IP 地理分布画像             │
    31|                    │  TTL: 90天滚动清理                  │
    32|                    └──────────────────────────────────┘
    33|```
    34|
    35|**关键区别：**
    36|- **Redis** = 原始明细数据，查"刚才发生了什么"
    37|- **Cassandra** = 聚合指标数据，查"这个账户长什么样"

    41|## 二、1000 TPS 瓶颈分析
    42|
    43|### 2.1 流量模型
    44|
    45|```
    46|1000 TPS = 每秒1000笔交易
    47|         = 每笔交易1ms处理窗口（理想）
    48|         = 实际：水平扩展后每笔~10ms ~ 50ms
    49|
    50|每秒数据量估算:
    51|  Kafka入站:   1000 msg/s × ~1KB = ~1MB/s
    52|  Redis读写:   1000 read + 1000 write/s
    53|  Cassandra读: 1000 read/s (cache miss时)
    54|  ONNX推理:    1000 inference/s
    55|```
    56|
    57|### 2.2 瓶颈矩阵
    58|
    59|| 组件 | 单节点能力 | 1000 TPS | 风险 | 缓解方案 |
    60||:---|:---:|:---:|:---:|:---|
    61|| **Kafka** | 100万+ msg/s | ✅ 轻松 | 无 | — |
    62|| **Redis** | 10万+ QPS | ✅ 轻松 | 🟢 低 | — |
    63|| **Cassandra** | 1万+ QPS/node | ✅ 可撑 | 🟢 低 | 适当副本数 |
    64|| **ONNX CPU** | 500-2000/s | ⚠️ 临界 | 🟡 中 | Batch推理 + GPU |
    65|| **ONNX GPU** | 5000-50000/s | ✅ 轻松 | 🟢 低 | T4/V100起步 |
    66|| **Janino规则** | 5万+ eval/s | ✅ 轻松 | 🟢 低 | 缓存编译结果 |
    67|| **Scanner** | 取决于复杂度 | ⚠️ 取决于实现 | 🟡 中 | 并行+超时熔断 |
    68|| **Quarkus** | Vert.x eventloop | ✅ 可撑 | 🟢 低 | 水平扩展 |
    69|
    70|### 2.3 核心瓶颈详解
    71|
    72|#### 🔴 风险最高的点：ONNX 模型推理
    73|
    74|```
    75|ONNX CPU推理延迟:
    76|  Gemm(1×10) + Sigmoid → ~0.01ms (微不足道)
    77|  如果模型复杂（树集成 / 深度网络）→ 可能0.5-5ms
    78|
    79|单核CPU:  1000ms/s ÷ 0.5ms = 2000 inference/s → OK
    80|多模型串联: 5个模型 × 0.5ms = 2.5ms → 400/s ❌
    81|```
    82|
    83|**方案：** 模型推理走 GPU（CUDA Execution Provider），延迟降到 ~0.05ms，吞吐提升 10-50x
    84|
    85|#### 🟡 需关注的点：Redis 热 key
    86|
    87|```
    88|热门账户的 key 可能被大量并发请求命中:
    89|  "user:9527:recent_txns" ← 100笔/秒同时读
    90|  
    91|Redis 单线程处理，热 key 会造成排队
    92|```
    93|
    94|**方案：** 本地缓存 + Redis 读写分离 + 热 key 打散
    95|
    96|#### 🟡 Scanner 超时雪崩
    97|
    98|```
    99|如果某个 Scanner 挂了（如外部设备指纹服务超时）:
   100|  → 整个 Pipeline 阻塞
   101|  → 请求排队
   102|  → 雪崩
   103|```
   104|
   105|**方案：** 每个 Scanner 设独立超时 + 熔断器（Resilience4j）+ fallback 策略
   106|
   107|### 2.4 优化后的架构
   108|
   109|```
   110|                         1000 TPS
   111|                            │
   112|                            ▼
   113|                    ┌──────────────┐
   114|                    │    Kafka     │ ← 10分区起
   115|                    └──────┬───────┘
   116|                           │
   117|              ┌────────────┼────────────┐
   118|              │  Consumer  │  Consumer  │  ... × N
   119|              ▼            ▼            ▼
   120|        ┌──────────┐ ┌──────────┐ ┌──────────┐
   121|        │Quarkus   │ │Quarkus   │ │Quarkus   │  ← 水平扩展
   122|        │Instance 1│ │Instance 2│ │Instance N│     (Kafka consumer group)
   123|        └────┬─────┘ └────┬─────┘ └────┬─────┘
   124|             │            │            │
   125|             ▼            ▼            ▼
   126|        ┌──────────────────────────────────────┐
   127|        │        Redis Cluster (6 nodes)        │
   128|        │    Raw transactions (24h TTL)         │
   129|        │    读写分离 + 本地缓存兜底             │
   130|        └──────────────────────────────────────┘
   131|             │            │            │
   132|             ▼            ▼            ▼
   133|        ┌──────────────────────────────────────┐
   134|        │     Cassandra Cluster (6 nodes)       │
   135|        │     Aggregated Metrics (90d TTL)      │
   136|        │     RF=3, 读写一致性 ONE              │
   137|        └──────────────────────────────────────┘
   138|             │            │            │
   139|             ▼            ▼            ▼
   140|        ┌──────────────────────────────────────┐
   141|        │   In-process ONNX (GPU)              │
   142|        │   Batch Inference: 32 per batch      │
   143|        │   → GPU: ~0.3ms/batch = ~10万 TPS    │
   144|        └──────────────────────────────────────┘
   145|             │            │            │
   146|             ▼            ▼            ▼
   147|        ┌──────────────────────────────────────┐
   148|        │   Decision + Result → Kafka/Database │
   149|        └──────────────────────────────────────┘
   150|```

   152|### 2.5 关键性能设计要点
   153|
   154|| # | 策略 | 说明 |
   155||:---|:---|:---|
   156|| 1 | **Reactive/异步** | Quarkus + Mutiny + Vert.x，非阻塞全链路 |
   157|| 2 | **Pipeline 并行化** | Scanner/Rule/Model 三路并行，CompletableFuture |
   158|| 3 | **ONNX Batch 推理** | 攒够32笔再送GPU推理，吞吐提升10x |
   159|| 4 | **Redis 本地缓存** | Caffeine 缓存热 key，减少 Redis 压力 |
   160|| 5 | **Cassandra 异步读** | 不阻塞 eventloop |
   161|| 6 | **超时 + 熔断** | 每个 stage 独立超时，失败走降级 |
   162|| 7 | **Kafka 分区数** | 分区 = consumer 数，建议 10-20 分区 |
   163|| 8 | **预热** | 启动时预加载热门账户特征到 Redis |
   164|
   165|### 2.6 容量估算总结
   166|
   167|```
   168|1000 TPS 架构完全可撑，但有几个前提:
   169|
   170|✅ Kafka:      1分区 = 100MB/s，10分区绰绰有余
   171|✅ Redis:      6节点 cluster，10万+ QPS，富余 100x
   172|✅ Cassandra:  6节点 cluster，RF=3，读 QPS 富余 10x
   173|✅ ONNX CPU:   刚好临界(~2000/s)，建议 GPU 起步
   174|✅ Quarkus:    N实例水平扩展，单实例 200-500 TPS
   175|
   176|🟢 结论: 架构本身没问题，注意模型推理走 GPU + 
   177|    为每个 Scanner 设超时熔断即可。
   178|```
